## 2. Curate — build a training set with filters

**Purpose.** Produce a curated `trainset.jsonl` of `PatternToRule` examples for
optimization. Composable filters let you slice by recency, emphasis, classification,
pattern, error type, and recovery signal.

**When to use.** Always run this before `sio optimize`. The default ground-truth
table is noisy; curate first, optimize second.

**Example — recommended starting point:**

```bash
sio curate \
  --since "30 days" \
  --emphasis \
  --classified \
  -o ~/.sio/datasets/trainset-2026-05-15.jsonl
```

This selects errors from the last 30 days where the user_message contains `!!` or
`??` (high-signal frustration) and which have already been classified by the
LLM router.

**Other useful slices:**

```bash
# Only one error type, with positive recovery signal:
sio curate --error-type tool_failure --has-positive-recovery -o tf.jsonl

# Pattern-scoped (e.g., all dbt-* patterns):
sio curate --pattern-prefix dbt- -o dbt.jsonl

# Exclude correction-driven rows (when you want pure errors, not user re-asks):
sio curate --since "7 days" --exclude-corrections -o pure_errors.jsonl

# Drop cascade-failure rows (sibling cancellations, not real bugs):
sio curate --since "30 days" --exclude-cascade -o no_cascade.jsonl
```

**Outputs.**
- `trainset.jsonl` — one `dspy.Example` per line
- `trainset.jsonl.preview.md` — human-readable summary (row counts, type mix, top patterns)

**Watch out for:**
- `--emphasis` is strict: it requires literal `!!` or `??`. If you get an empty
  set, drop it before assuming the filter is broken.
- `--classified` excludes anything from before today (when classify-on-mine
  shipped). Use `--since "1 day"` to stay safely in the classified era.

**Companions.** `sio optimize --trainset-file`, `sio analyze same-error`.

## 3. Amplify — synthetic dataset expansion (FLAGGED)

**Purpose.** Take a curated trainset and produce N variants per row via Gemini
Flash, judged for quality.

**When to use.** **Don't, yet, for production runs.** The v1 home-grown
amplifier regressed end-to-end. Only use today with `--baseline-against` on
`sio optimize` so the regression gate catches the bad output before it lands.

**Example (regression-gated only):**

```bash
sio amplify \
  --input ~/.sio/datasets/trainset-2026-05-15.jsonl \
  --output ~/.sio/datasets/trainset-amp.jsonl \
  --n-per-row 3 \
  --min-judge-score 0.7 \
  --max-workers 4
```

Then optimize with a baseline guard:

```bash
sio optimize --trainset-file ~/.sio/datasets/trainset-amp.jsonl \
             --baseline-against <prior_module_id>
```

If the new module scores below the baseline, it is marked inactive automatically.

**Watch out for:**
- Cost: one Gemini call per row for generation, one per row for judging.
- Quality regressions: see the PRD note above. Real fix is distilabel; see
  `~/dev/prd/scratch/synthetic_amplification_distilabel_2026-05-15.md`.

**Companions.** `sio curate` (input), `sio optimize --baseline-against` (safety net).

## 4. Optimize — DSPy with explicit trainset + regression gate

**Purpose.** Run the DSPy optimizer (GEPA by default) against a curated trainset,
with a regression gate against a known-good prior module.

**When to use.** After `sio curate`. Always pass `--baseline-against` once you
have any optimized module on record.

**Example:**

```bash
sio optimize \
  --trainset-file ~/.sio/datasets/trainset-2026-05-15.jsonl \
  --baseline-against <prior_module_id>
```

If the new module scores below the prior, SIO stores it with `active=False` so
the existing module continues to serve traffic.

**Optimizer selection** (existing — unchanged today):

```bash
sio optimize --optimizer gepa       # default
sio optimize --optimizer mipro      # MIPROv2
sio optimize --optimizer bootstrap  # BootstrapFewShot
```

**Watch out for:**
- All LM construction goes through `sio.core.dspy.lm_factory`. Never instantiate
  `dspy.LM()` directly in tests or scripts.
- A score of exactly `1.0` is the classic template-fallback signal — `sio doctor`
  now flags this as unrealistic.

**Companions.** `sio curate` (input), `sio doctor` (verify module health),
`sio apply --auto-threshold` (ship the rules).

## 5. Apply — bulk apply pending suggestions

**Purpose.** Approve and apply pending suggestions in one shot, gated by score
and de-duplicated against existing rules.

**When to use.** After review, when you trust the optimized module's output.

**Example:**

```bash
sio apply --auto-threshold 0.85 --skip-dupes
```

- `--auto-threshold 0.85` — only apply suggestions with score ≥ 0.85
- `--skip-dupes` — skip suggestions whose rule body is a near-match to an existing rule

**Watch out for:**
- This writes to `CLAUDE.md` / `~/.claude/rules/`. Use `--dry-run` (existing flag)
  on the first pass.
- Lower the threshold cautiously — 0.7 is roughly the noise floor on the current
  optimized module.

**Companions.** `sio review` (interactive review), `sio velocity --by-rule`
(check that applied rules actually help).

## 6. Validate — doctor, velocity, same-error

### `sio doctor`

**Purpose.** End-to-end health check. Today gained a DSPy pipeline check that
verifies an active `suggestion_generator` module exists and that its score is
realistic (not the trivial `1.0` template-fallback marker).

**Example:**

```bash
sio doctor
```

**Watch out for:**
- A `1.0` score in the DSPy check means the optimizer fell back to a template
  and never ran. Re-run optimize with a non-empty trainset.

### `sio velocity --by-rule`

**Purpose.** Per-rule attribution of error-rate change. Reads the
`error_records.active_rules` JSON column captured at mine time.

**Example:**

```bash
sio velocity --by-rule --min-records 50
```

Returns each rule's best/worst delta by error_type. Use `--min-records` to
suppress noisy rules with too little data.

**Watch out for:**
- Only rows mined AFTER today (when `active_rules` started being stamped) have
  attribution data. Older rows are silently excluded.

### `sio analyze same-error`

**Purpose.** Find error signatures that repeat ≥N times across sessions — i.e.,
the agent is failing to learn.

**Example:**

```bash
sio analyze same-error --min-count 3 --since "14 days" --with-context
```

- `--min-count` — signature must appear at least N times
- `--with-context` — include surrounding turns for triage

**Watch out for:**
- "Same error" is signature-based, not exact-string. Tune via the classifier if
  unrelated errors are bucketing together.

**Companions.** `sio curate --has-positive-recovery` (find the cases that
*did* recover, to learn from).

## 7. Diagnose — `sio differential-flows`

**Purpose.** Twin-flow finder: surface tool sequences that look identical up to
a point but diverge into success vs failure. The most actionable signal SIO
produces.

**When to use.** When `velocity` shows a regression but `analyze same-error`
doesn't pinpoint why. Differential flows show the *divergence point*.

**Example:**

```bash
sio differential-flows \
  --min-success 5 \
  --min-failure 3 \
  --per-cohort 10 \
  --max-hashes 200 \
  -o ~/.sio/datasets/diffs.jsonl
```

**Feed positives directly into the dataset builder:**

```bash
sio differential-flows \
  --min-success 5 --min-failure 3 \
  --positives-for-builder
```

This emits canonical `PatternToRule` rows — usable as input to `sio curate` or
straight into `sio optimize --trainset-file`.

**Watch out for:**
- High `--per-cohort` and `--max-hashes` quickly blow up runtime. Start small.

**Companions.** `sio curate`, `sio optimize`.

## 8. Promote — gold standards and positive recoveries

### `sio promote-to-gold`

**Purpose.** Promote a `behavior_invocations` row (or all eligible) to
`gold_standards`. Until today this was a phantom command (CLI shell, no
implementation).

**Examples:**

```bash
# Promote one:
sio promote-to-gold <invocation_id>

# Bulk:
sio promote-to-gold --all-eligible

# Preview:
sio promote-to-gold --all-eligible --dry-run
```

**Watch out for:**
- "Eligible" criteria are encoded in `src/sio/cli/main.py` around line 2407 —
  check there if a row you expect doesn't promote.

### `sio promote-positives`

**Purpose.** Pair `positive_records` with the immediately preceding error and
write the pair into `ground_truth(status=pending)`. This is how a recovery
becomes a candidate teaching example.

**Example:**

```bash
sio promote-positives --since "7 days" --min-confidence 0.7
sio promote-positives --since "7 days" --dry-run
```

**Watch out for:**
- Status is `pending`, not approved — you still need `sio review` to admit them
  into the live ground truth.

**Companions.** `sio review`, `sio curate --has-positive-recovery`.

## Canonical Daily Loop

```bash
# 1. See what's new
sio doctor
sio velocity --by-rule --min-records 50

# 2. Build a fresh trainset
sio curate --since "7 days" --emphasis --classified \
  -o ~/.sio/datasets/trainset-$(date +%F).jsonl

# 3. Optimize with a regression gate
sio optimize \
  --trainset-file ~/.sio/datasets/trainset-$(date +%F).jsonl \
  --baseline-against <last_active_module_id>

# 4. Review + apply
sio review
sio apply --auto-threshold 0.85 --skip-dupes

# 5. If something looks off, diagnose
sio analyze same-error --min-count 3 --since "7 days"
sio differential-flows --min-success 5 --min-failure 3 -o /tmp/diffs.jsonl
```
