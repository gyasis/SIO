Module 1

What SIO Does — Catching Your AI in the Act

A flight data recorder for your AI agent. After every flight, it pulls the black box, finds the moments your copilot stalled, and writes a new line in the manual so the same maneuver never bites you twice.

Your AI just lied to you. Again.

Your AI agent confidently ran sed -i to fix one line in your .env file. It silently wiped the file. Last month it did the exact same thing on a different machine.

You wrote a note. You told it not to. And yet — next session, fresh context, same mistake.

⚠️
The forgetting problem

Every new session, your AI starts with no memory of yesterday’s screw-ups. The only thing it remembers is whatever you put in its CLAUDE.md file. So who’s writing those rules? You. By hand. Every time.

SIO — the Self-Improving Organism — is the system that writes those rules for you, drafted from the actual mistakes your agent has already made.

The flight data recorder

Commercial airliners carry a black box that records every control input, every warning, every gust of wind. After a rough landing, investigators don’t guess — they read the tape.

SIO is that black box for your AI. A tiny hook watches every tool call your agent makes, copies the receipt into a tiny database, and lets a learning pipeline do the rest after the flight.

The pipeline, end to end

1
Sessions

Raw recordings of every tool call your agent made — the input, the output, the error if there was one.

2
Mine

Sift the recordings for failures, user corrections, repeated attempts — the moments the agent stumbled.

3
Cluster

Group similar stumbles together. “Wiped a file with sed” and “wiped a file with perl -pi” are the same lesson.

4
Rank

Patterns that hurt the most or recur the most float to the top. Rare flukes sink.

5
Suggest

Draft a plain-English rule for each top pattern — the kind of thing you’d hand-write into CLAUDE.md.

6
Review

You read the draft. Approve it, edit it, or throw it out. SIO never writes without your say-so.

7
Apply

Approved rules land in CLAUDE.md. Next session, the agent reads them. Same mistake — blocked.

🔁
This is a closed feedback loop

The agent’s output becomes input to a learner, the learner’s output becomes input to the agent. Engineers call this a feedback loop. The only part that isn’t automatic is you, the approver in the middle. That’s on purpose.

The whole thing starts with one tiny function

Every time your AI calls a tool, Claude Code fires an event called PostToolUse. SIO listens for that event with the function below. It’s 30 lines. That’s the entire data-collection surface.

CODE 
"""PostToolUse hook handler — captures telemetry from Claude Code tool calls."""

from __future__ import annotations
import json, logging, os, sys
from datetime import datetime, timezone

logger = logging.getLogger(__name__)
from sio.core.constants import DEFAULT_PLATFORM as _DEFAULT_PLATFORM
_DEFAULT_DB_DIR = os.path.expanduser(f"~/.sio/{_DEFAULT_PLATFORM}")


def handle_post_tool_use(stdin_json: str, *, conn=None) -> str:
    """Process a PostToolUse hook event.

    Parses the JSON payload, logs the invocation to the database,
    and always returns {"action": "allow"}.
    """
    try:
        payload = json.loads(stdin_json)
    except (json.JSONDecodeError, TypeError):
        return json.dumps({"action": "allow"})

    own_conn = conn is None
    if own_conn:
        db_path = os.path.join(_DEFAULT_DB_DIR, "behavior_invocations.db")
        os.makedirs(_DEFAULT_DB_DIR, exist_ok=True)
        conn = init_db(db_path)

    session_id = payload.get("session_id", "unknown")
    tool_name  = payload.get("tool_name", "unknown")
    tool_input = json.dumps(payload.get("tool_input", {}))
    tool_output = payload.get("tool_output", "")
    error = payload.get("error")
    user_message = payload.get("user_message", "[UNAVAILABLE]")

    log_invocation(conn=conn, session_id=session_id, tool_name=tool_name,
                   tool_input=tool_input, tool_output=tool_output,
                   error=error, user_message=user_message,
                   platform=_DEFAULT_PLATFORM)

    return json.dumps({"action": "allow"})
PLAIN ENGLISH

Every time your AI uses a tool, Claude Code hands this function a little envelope of telemetry: which tool, what input, what came back, and whether it errored.

If the envelope is garbage and can’t even be parsed, the function shrugs and says allow. It never blocks the agent.

Otherwise it opens a tiny SQLite file at ~/.sio/claude-code/behavior_invocations.db and writes one new row: session ID, tool name, input, output, error, the last thing you said.

Then it returns {"action": "allow"} — unconditionally. SIO is a passive observer. The agent never even notices it was watched.

💡
SIO is a passive observer, not a guard

Hooks never block. They only watch and write. The agent stays fast; learning happens offline. If SIO itself ever crashes, your agent keeps right on working — it just stops getting smarter that day.

Why the loop only closes through you

Lots of "self-improving" systems sound dangerous — an AI that rewrites its own rules with no human in the way. SIO is built differently. The hook captures freely, the pipeline drafts freely, but nothing reaches CLAUDE.md until you click approve.

👁️
Capture is automatic

The Hook runs on every tool call. No prompts, no approvals. Cheap and silent.

🔬
Mining is automatic

The Miner re-reads the database whenever you run sio scan. Failures get clustered into patterns.

✍️
Suggestions are automatic

For each ranked pattern, the Suggester drafts a plain-English rule. You see the draft, not the rule.

✅
Approval is manual — always

You read each suggestion in sio review. Approve, edit, or kill. The agent never writes its own constitution.

🧠
Your CLAUDE.md is the agent’s long-term memory

Every session, the agent re-reads it before doing anything else. A rule written today is in effect tomorrow — on every machine that syncs the file. That’s why drafting rules is worth doing carefully, and why SIO bothers to involve you instead of just patching the file behind your back.

What the Miner Actually Reads For

Every actor's job is documented in the docstring at the top of its main file. The Miner opens with this — it tells you exactly which five things it watches for in your transcripts.

CODE 
"""Error extractor — classifies parsed conversation messages into five error
categories and emits ErrorRecord dicts suitable for insertion into the v2
``error_records`` table.

Error types detected
--------------------
tool_failure       — assistant message whose ``error`` field is non-null
user_correction    — human message containing correction phrasing
repeated_attempt   — same tool_name called 3+ consecutive times with similar input
undo               — human message containing undo / revert signals
agent_admission    — assistant message where the AI admits a mistake
"""
PLAIN ENGLISH

The Miner reads your chat transcripts and tags any message that smells like trouble.

It only cares about five flavors of trouble — anything else gets ignored.

tool_failure — the AI ran a command and it crashed.

user_correction — you typed "no, that's wrong" (or anything that sounds like it).

repeated_attempt — the AI tried the same thing three times in a row. Definitely stuck.

undo — you asked the AI to revert or roll back.

agent_admission — the AI itself said "oops, I was wrong."

What the Clusterer Does Next

Once the Miner hands over a pile of tagged errors, the Clusterer groups the ones that mean the same thing — even if the words are different. It uses an embedding for each error and measures cosine similarity.

CODE 
"""Embedding-based error pattern clusterer.

Groups a list of error record dicts into semantic clusters using cosine
similarity on fastembed embeddings.  The public API is a single function:

    cluster_errors(errors, threshold=0.70) -> list[dict]
"""
PLAIN ENGLISH

This file's only job: take a bag of errors and sort them into themed piles.

It uses a math trick to ask "are these two errors saying the same thing?" — even if the wording differs.

The whole feature is exposed as one function — feed it errors, get back clusters.

The default threshold (0.70) is how picky it is. Higher = tighter, more piles. Lower = looser, fewer piles.

The Applier Is the Only Actor That Touches Disk

Everything upstream — Hook, Miner, Clusterer, Suggester, Reviewer — is read-only or talks to a database. Only the Applier can edit your actual config files. And before it does, it checks one thing.

CODE 
_ALLOWED_ROOTS: list[Path] = [
    Path.home() / ".sio",
    Path.home() / ".claude",
]

def _validate_target_path(path: Path, *, extra_roots: tuple[Path, ...] = ()) -> str | None:
    """Return an error message if path is outside allowed roots or cwd."""
    resolved = path.resolve()
    allowed = (*_ALLOWED_ROOTS, Path.cwd(), *extra_roots)
    for root in allowed:
        try:
            resolved.relative_to(root.resolve())
            return None
        except ValueError:
            continue
    return f"Target path {resolved} is outside allowed directories"
PLAIN ENGLISH

There's a short list of folders SIO is allowed to write inside: its own ~/.sio/ and the Claude config folder ~/.claude/. That's it.

Before SIO writes anything to disk, it asks: is the target file inside one of these allowed folders (or the folder you're currently in)?

It tries each allowed folder in turn. If the path fits inside one, it's safe — return "no error."

If none of them match, refuse the write and return an error message instead.

This is the seatbelt. Even a buggy Suggester writing nonsense rules can't trash /etc/passwd because the Applier won't let it.

🛡️
The Applier is the only actor with write permission to your config.

Everything upstream is read-only or database-only. This is intentional — engineers call it a narrow blast radius. When you see a permission error from SIO, you almost always look in src/sio/applier/ first.

Panning for gold in a river of chatter

The five shapes of "rock"

The Miner scans each parsed message and asks one question: does this match one of five known error shapes? If yes, emit an ErrorRecord. If no, drop it.

CODE 
"""Error extractor — classifies parsed conversation messages into five error
categories and emits ErrorRecord dicts.

Error types detected
--------------------
tool_failure       — assistant message whose ``error`` field is non-null
user_correction    — human message containing correction phrasing
repeated_attempt   — same tool_name called 3+ consecutive times with similar input
undo               — human message containing undo / revert signals
agent_admission    — assistant message where the AI admits a mistake
                    (e.g. "I made a mistake", "I should have", "my apologies")
"""
PLAIN ENGLISH

tool_failure — the AI tried to do something and the tool returned an error (a failed shell command, a 404, a syntax error).

user_correction — you typed "no, actually…" or "that's wrong." A human-supplied course-correction.

repeated_attempt — the AI called the same tool three times in a row with nearly-identical input. A retry loop.

undo — you asked it to revert or back out. A stronger correction than a verbal "no."

agent_admission — the AI itself said "my apologies" or "I made a mistake." Self-reported failure.

💡
No LLM has read your code yet

Detection is pure pattern matching — regex plus structural inspection of the JSON. SIO can mine months of history offline, on a plane, with the API down. The LLM only enters in the next module.

Inside a detector: user_correction

Each error type is just a list of regex patterns. The Miner walks every human message and checks each pattern. One match → emit an ErrorRecord. Here's the real list:

CODE 
_CORRECTION_PATTERNS: list[re.Pattern[str]] = [
    re.compile(r"\bno,?\s+actually\b", re.IGNORECASE),
    re.compile(r"(?:^|(?<=\s)|\A)no,", re.IGNORECASE),
    # ... "that's wrong", "I meant", "stop", "undo that", etc.
]
PLAIN ENGLISH

Make a list called _CORRECTION_PATTERNS. Each item is a compiled regex.

First pattern matches "no actually" or "no, actually" anywhere in the message. Case-insensitive.

Second pattern matches a "no," at the very start of a message or after whitespace.

More patterns follow — "that's wrong," "I meant," "stop," "undo that."

The Miner does not understand what you were correcting. It only notices the shape of correction language. That's enough — and it's why this stage is so fast.

The Clusterer's contract

Once the Miner has produced a stack of raw errors, the Clusterer's job is to group ones that mean the same thing. Its public API is a single function: feed it errors, get back patterns.

CODE 
"""Groups a list of error record dicts into semantic clusters using cosine
similarity on fastembed embeddings.  The public API is a single function:

    cluster_errors(errors, threshold=0.70) -> list[dict]

Pattern dict schema
-------------------
pattern_id   : str   — centroid-hash slug (format: ``<top_term>_<hex10>``)
description  : str   — representative error text (first error's text)
error_count  : int   — number of errors in the cluster
session_count: int   — number of distinct session_ids in the cluster
rank_score   : float — 0.0 initially; downstream ranker sets this
"""
PLAIN ENGLISH

Call cluster_errors with a list of errors. Anything with cosine similarity above 0.70 ends up in the same pile.

Each output pattern gets a unique id built from its top term plus a short hash.

It carries a representative error text (the first one in the cluster) so you have something human-readable to read.

It counts how many errors landed in this pile, and how many distinct sessions they came from.

rank_score starts at zero. A later stage (the Ranker) will sort patterns by importance.

The magic word is threshold=0.70. SIO uses fastembed to turn each error's text into a vector, then asks: is this vector within 0.70 of any cluster's center? Yes → join the cluster. No → start a new one.

Storing the centroid — a small but clever trick

Every cluster has a centroid: the average vector of all its errors. SIO packs that centroid into a compact BLOB and saves it in SQLite.

CODE 
def _pack_centroid(vec: np.ndarray, model_hash: bytes) -> bytes:
    """Pack vec into the R-9 BLOB format.

    Format: [dim: uint32_le (4 bytes)] [model_hash: 8 bytes] [vector: float32[dim]]
    """
    if len(model_hash) != 8:
        raise ValueError(f"model_hash must be exactly 8 bytes, got {len(model_hash)}")
    dim = len(vec)
    header = struct.pack("<I", dim) + model_hash
    floats = vec.astype(np.float32).tobytes()
    return header + floats
PLAIN ENGLISH

Take the cluster's average vector and a short hash identifying which embedding model produced it.

First 4 bytes: the vector's dimension (e.g. 384). Next 8 bytes: the model hash so we know if we ever swap models.

Safety check: the model hash must be exactly 8 bytes, no more, no less.

Then dump the raw floating-point numbers as bytes. Done. One row in SQLite holds the whole centroid.

💡
The centroid is your savings account

Once a centroid is stored as a BLOB, every new error compares against the saved vector. SIO never re-embeds the whole corpus to ask "is this familiar?" That's how the same script can run nightly on years of history without blowing up.

Shift #1 — Prompts are typed contracts

Most people think of a prompt as a paragraph you paste into ChatGPT. DSPy disagrees. A prompt is a signature: inputs in, outputs out, docstring tells the LLM how.

CODE 
class PatternToRule(dspy.Signature):
    """Generate a concise CLAUDE.md rule that prevents the given error pattern.
    The rule must be actionable, file-path-safe, and <= 3 sentences.

    CRITICAL — required specificity (B5 grounding directives):
    - The rule TITLE must reference the SPECIFIC tokens from `pattern_description`
      (tool name, env var, path, command — whatever recurs in `Common phrases:`).
      Generic titles like "Always check inputs" or "Verify configuration" are
      WRONG — they lose the discriminating signal that justifies the rule.
    - The rule BODY must cite the concrete failure observed in `example_errors`.
    - If `example_errors` shows an env var, file path, or command literal,
      that literal MUST appear verbatim in the rule.
    """
    # input fields: pattern_description, example_errors, project_context
    # output fields: rule_title, rule_body, rule_rationale
PLAIN ENGLISH

A Signature is a typed contract for an LLM. Inputs go in, outputs come out, and the docstring tells the LLM how.

The "B5 grounding directives" are explicit anti-platitude rules: don't say "always check inputs" — name the actual command (sed -i, $ZENO_DIR, whatever).

The docstring itself is the prompt. There's no separate prompt template file. This is the whole point of DSPy — and it's why this docstring is so detailed.

DSPy fills in the inputs (the pattern, the examples), sends the whole thing to the LLM, and parses the three outputs back out.

💡
Aha — The docstring IS the prompt

When you change a Signature's docstring, you've changed how the LLM behaves. There is no separate "prompt.txt" file anywhere. That's the entire point of DSPy: prompts live as typed code, not as throwaway strings.

One door, every LLM call walks through it

Across the whole SIO codebase, there is exactly one function that constructs an LLM. Every call routes through it. A grep test in the test suite literally fails the build if anyone bypasses it.

CODE — lm_factory.py 
"""LM backend factory — single-source dspy.LM construction (FR-041, SC-022).

All dspy.LM(...) construction happens here. No other file in src/sio/ may
construct dspy.LM directly; the grep test in test_lm_factory.py enforces this.

Environment overrides:
  SIO_TASK_LM       — model string for get_task_lm()  (default: openai/gpt-4o-mini)
  SIO_REFLECTION_LM — model string for get_reflection_lm() (default: openai/gpt-5)
"""
import os, dspy, litellm
litellm.drop_params = True

def get_task_lm() -> dspy.LM:
    """LM used for normal module forward passes. Cheap, fast, cached."""
    model = os.environ.get("SIO_TASK_LM", "openai/gpt-4o-mini")
    return dspy.LM(model, cache=True, temperature=0.0, max_tokens=4096)
PLAIN ENGLISH

Every LLM call in SIO routes through get_task_lm(). Why?

So you can swap models with one environment variable: SIO_TASK_LM=anthropic/claude-3-5-sonnet — done. No code changes.

And a single grep test enforces it: nobody can sneak in a direct dspy.LM(...) call elsewhere — the build breaks.

litellm.drop_params = True says: if a model doesn't support a parameter, silently drop it instead of crashing. Makes provider swaps painless.

Single point of LLM construction = single point of safety.

Shift #2 — Prompts are optimizable

Here's the part that breaks most people's mental model: in DSPy, you can train the prompt. Not the model — the prompt itself.

⚡
Aha — Optimizers don't fine-tune the model. They fine-tune the prompt.

No model weights are updated. DSPy runs your module against a corpus of labeled examples, mutates the prompt (rewording instructions, adding few-shot examples), and keeps the version that scores highest. The result is a JSON file at ~/.sio/optimized/ — a "trained prompt" you can ship.

SIO ships with three optimizers. You pick one with a flag: sio optimize --optimizer gepa.

GEPA (default)

Reflective. Uses a second, smarter LM to critique the first one's output and suggest a better prompt. Slow but best quality. The default for SIO.

Bayesian search over instruction variants. Good when you have 50+ labeled examples to score against. Methodical, broad coverage.

Few-shot bootstrapping. Generates synthetic examples from your trainset and embeds the best ones in the prompt. Fast, works with minimal data.

Module 5

Closing the Loop — Apply Safely or Not At All

The three gates between a generated suggestion and a real change on disk.

🩺

The Surgical Timeout

Before any operation, the surgical team pauses. They read the chart aloud, confirm the procedure, confirm the site. Only then does the first cut happen — and a copy of the patient's pre-op state is filed away, kept for ten patients back.

If something goes wrong, the surgeon can rewind to before the cut.

SIO's Applier works the same way. It never writes without a backup, never writes outside permitted directories, and never writes a rule the human didn't approve.

A rule has been generated. Now what?

The DSPy brain just produced this string and dropped it in the suggestions table:

"Never use sed -i — use the Edit tool instead."

Nothing has touched your CLAUDE.md yet. So what's the path from "suggestion in a database" to "rule in your config file" — and how does SIO make sure that path is reversible?

Three gates. Human approval. Path validation. Atomic backup. Each one lives in its own file. Each one alone could stop a bad write.

Gate 1 — The approval status check

Before the Applier touches disk, it checks one field on the suggestion row: status. If the status isn't approved or auto_approved, the function bails.

def apply_change(db, suggestion_id, config=None, force=False) -> dict:
    row = db.execute(
        "SELECT * FROM suggestions WHERE id = ?", (suggestion_id,)
    ).fetchone()

    if row is None:
        return {"success": False, "reason": "Suggestion not found"}

    suggestion = dict(row)

    if suggestion["status"] not in ("approved", "auto_approved"):
        return {
            "success": False,
            "reason": f"Suggestion is not approved (status: {suggestion['status']})",
        }
    # ... budget check, similarity merge, atomic write with backup

Look up the suggestion by ID. If it doesn't exist, return a failure dict — no exception, no surprise.

Read the status field. If it's anything other than approved or auto_approved, refuse the operation.

A suggestion in pending state cannot be applied even if you pass its ID directly to sio apply. The human-in-the-loop gate is enforced in code, not just in the CLI.

Gate 2 — The path allowlist

Even an approved suggestion can't write just anywhere. The Applier validates the target path against a tiny allowlist: your home ~/.sio, your home ~/.claude, or your current working directory.

_ALLOWED_ROOTS: list[Path] = [
    Path.home() / ".sio",
    Path.home() / ".claude",
]

def _validate_target_path(path: Path, *, extra_roots: tuple[Path, ...] = ()) -> str | None:
    """Return an error message if path is outside allowed roots or cwd."""
    resolved = path.resolve()
    allowed = (*_ALLOWED_ROOTS, Path.cwd(), *extra_roots)
    for root in allowed:
        try:
            resolved.relative_to(root.resolve())
            return None
        except ValueError:
            continue
    return (
        f"Target path {resolved} is outside allowed directories: "
        f"{', '.join(str(r) for r in allowed)}"
    )

This is the kill switch. resolve() turns any sneaky path — like ../../etc/passwd — into an absolute path. No tricks.

Then we walk through each allowed root and ask: is the target inside this folder? If yes, return None (meaning "no error, you may proceed").

If nothing matches, return an error string. The Applier will not write if this returns anything but None. No matter how the suggestion's target_file got into the database, this seatbelt catches it before disk.

This defends against path traversal. Even if a generated rule somehow contained a hostile target, it can't write outside the sandbox.

Gate 3 — Budget check + atomic backup

Once approval and path pass, two final checks: is there room in the file, and is the old version safely copied aside?

"""sio.applier.budget -- instruction budget management for config files.

Public API
----------
    count_meaningful_lines(file_path) -> int
    check_budget(file_path, new_rule_lines, config) -> BudgetResult
    trigger_consolidation(file_path, config) -> bool
"""

class BudgetResult(NamedTuple):
    status: str  # 'ok' | 'consolidate' | 'blocked'
    current_lines: int
    cap: int
    message: str

Every instruction file has a budget cap. Before adding a rule, SIO counts the meaningful lines in your CLAUDE.md.

Three outcomes: ok (write it), consolidate (merge similar rules first), or blocked (refuse — you're at capacity).

Then the atomic write happens: the old file is copied to ~/.sio/backups/ with a timestamp, then the new content is written. The last 10 backups are kept.

Three gates, all enforced in code. Human approval lives in writer.py. Path validation lives in writer.py. Atomic backup lives in writer.py. Even if one gate had a bug, the others would still defend the system.

Every apply is reversible

The Applier writes a row to the applied_changes table for every successful write — including the diff before and after. That's what makes the undo button work.

"""sio.applier.rollback — revert applied changes.

Public API
----------
    rollback_change(db, change_id) -> dict
"""

One function. One job. rollback_change takes a change ID, looks up the stored diff, and restores the file to its pre-write state.

You can experiment fearlessly: try an apply, look at the result, run sio apply --rollback <id> if you don't like it. The pre-write snapshot is already in the database.

sio apply is reversible. The backup folder is your time machine. The applied_changes table is the index telling you which file to put back where.
